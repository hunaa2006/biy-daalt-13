# ADR (Architectural Decision Record)

### Шийдвэрийн гарчиг: Төслийн бүтэц болон өгөгдлийн сангийн сонголт

### Статус: Хэрэгжүүлсэн (Accepted)

### Контекст:
"Personal Task Tracker" төсөлд өгөгдлийг найдвартай хадгалах, хялбар удирдах, мөн кодыг цаашид өргөжүүлэхэд ойлгомжтой байх бүтэц шаардлагатай байсан. Үүнд дараах хоёр гол сонголт байсан:

1. Бүх өгөгдлийг түр санах ой (Memory)-д хадгалах уу, эсвэл өгөгдлийн сан (Database) ашиглах уу?

2. Кодоо нэг файл дотор бичих үү, эсвэл давхаргаар (Service-Oriented) салгах уу?

### Шийдвэр:

1. **Өгөгдлийн сан**: SQLite (better-sqlite3) ашиглахаар шийдсэн. Учир нь энэ нь файл дээр суурилдаг тул суулгахад хялбар, мөн сервер унтарсан ч өгөгдөл устахгүй байх давуу талтай.

2. **Бүтэц**: Service-Oriented Architecture-ыг сонгосон. Ингэснээр routes (замууд), services (бизнес логик), database (холболт) гэж салгасан.

### Үр дүн:

**Давуу тал**: Код маш цэгцтэй болсон. Алдаа гарлаа гэхэд хаана байгааг нь (жишээ нь SQL алдаа бол Service-ээс, замын алдаа бол Route-ээс) хурдан олох боломжтой болсон.

**Сул тал**: Жижиг төсөлд хэтэрхий олон файл үүсгэх шаардлага гарсан боловч энэ нь цаашид төсөл томорход тустай.
 ### AI-ийн нөлөө:
  
Анх би бүх кодоо app.js дотор бичих гэж байсан боловч AI-тай зөвлөлдсөний үндсэн дээр taskService гэж тусад нь салгасан. Энэ нь кодын уншигдах байдлыг сайжруулсан.
